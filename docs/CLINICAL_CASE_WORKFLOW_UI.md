# 临床病例工作流 UI 重构方案

本文记录下一轮 UI 层重构的产品化方向：从按工具/技术模块组织的工作台，转向按病例任务组织的临床向导式工作流。目标是让医生围绕一个病例完成评估、定位、规划、模拟和输出，而不是面对一组研发工具自行拼流程。

## 目标

- 医生主流程只暴露临床任务：面部评估、病灶定位、切口规划、闭合模拟、方案确认。
- 标注器、图谱库、Provider 配置、R3F 预览、资产加载和工程诊断从医生主流程移出。
- 现有功能以隐藏、重组、嵌入为主，不直接删除，避免破坏研发、科研和回归测试路径。
- 页面文案去技术化，不向医生显示 R3F、FLAME、MediaPipe、Agent trace、topology 等工程术语。
- 用步骤条和病例状态代替左侧工具堆叠，明确每一步的输入、完成条件和下一步动作。

## 当前功能重新定位

| 当前路径 / 功能 | 当前状态 | 产品化定位 |
| --- | --- | --- |
| `/app/live` 实时 RSTL 显示 | Stage 1 主路径 | 纳入“面部评估与张力线映射”，作为病例向导第一步 |
| `/app/incision` 切口候选工作台 | Stage 2 主路径 | 纳入“病灶定位与切口规划”，作为病例向导第二步 |
| `/app/surgery` 沿 RSTL 闭合演示 | 独立演示页 | 降级为“闭合张力模拟”功能按钮/抽屉，嵌入切口规划步骤 |
| `/app/annotate` 3D 线标注 | 图谱生产/复核工具 | 移出医生主流程，放入系统设置或图谱库管理 |
| `/app/three-preview` R3F 标准脸预览 | 技术验证页 | 默认隐藏，只保留系统诊断入口 |
| LLM Provider 配置 | 远端 OpenAI-compatible / vLLM 连通性测试 | 移到全局设置或开发者配置，不放在医生主画布旁 |
| 浏览器 workflow trace / Agent trace | 审计与工程诊断 | 默认隐藏在“规划依据 / 审计记录”里，必要时展开 |
| FLAME / MediaPipe / topology 状态 | 资产和拓扑工程状态 | 转译为“3D 面部模型 / 面部追踪 / 图谱匹配状态” |

## 信息架构

产品化后建议分成三类入口：

| 类型 | 面向对象 | 放置位置 |
| --- | --- | --- |
| 临床主流程 | 医生 | 病例大厅和病例向导 |
| 科研/管理工具 | 研究员、管理员 | 系统设置、图谱库管理、模型/规则配置 |
| 工程诊断 | 开发者 | 隐藏入口、诊断面板、DevTools、审计导出 |

建议新增或重组为以下路由层级：

| 路由建议 | 职责 |
| --- | --- |
| `/app/cases` | 病例大厅，显示新建病例、导入病例、草稿和历史记录 |
| `/app/case/new` | 新建病例入口，收集基础病例信息和前置参数 |
| `/app/case/:id/evaluate` | 面部评估与张力线映射 |
| `/app/case/:id/plan` | 病灶定位、候选切口规划和闭合模拟 |
| `/app/case/:id/review` | 方案确认、报告导出和实时叠加输出 |
| `/app/settings/atlas` | 图谱库管理，承载当前 3D 标注工具的受控入口 |
| `/app/settings/developer` | AI 服务配置、资产诊断和隐藏技术预览的受控入口 |

现有 `/app/live`、`/app/incision`、`/app/annotate`、`/app/surgery` 可以先保留为兼容入口，不需要在第一轮重构中删除；但 `/app/settings/atlas` 和 `/app/settings/developer` 不应再直接 redirect 到旧工具页，而应先呈现设置壳、说明边界，再提供受控入口。

## 病例向导流程

### 阶段 0：病例大厅

定位：医生进入系统后的工作台首页，不直接进入人脸画布。

主要内容：

- 新建面部评估 / 导入新病例。
- 最近病例、草稿、已确认方案。
- 每个病例显示状态：待评估、待标记病灶、待确认、已导出。
- 全局设置入口：图谱库管理、AI 服务配置、系统诊断。

设计原则：

- 不展示 R3F、FLAME、MediaPipe、Provider 等工程概念。
- 不把科研标注工具放在医生主按钮区。
- “继续上次病例”优先于“选择技术路线”。

### 阶段 1：面部评估与张力线映射

定位：确认当前病例的影像或 3D 输入足够支撑张力线显示。

布局建议：

- 左侧：病例基础信息、年龄分档、影像来源、采集方式。
- 中央：照片、视频、摄像头或 3D 面部画布。
- 右上浮层：图层开关，例如基础张力线、线密度、透明度、个性化皮纹追踪。
- 底部或右下：主按钮“下一步：标记病灶”。

完成条件：

- 已上传照片/视频、打开摄像头，或完成可用的 3D 面部输入。
- 面部追踪状态达到可用。
- 张力线映射成功，或系统明确提示需要重新采集。

从现有功能复用：

- `/app/live` 的上传、摄像头、暂停、导出、线密度、透明度、镜像、追踪质量。
- 切口实时叠加 QA 继续保留，但默认转译为“方案叠加状态”。

### 阶段 2：病灶定位与切口规划

定位：把关注点从全脸切换到局部病灶和候选切口。

布局建议：

- 顶部：步骤条和病例摘要，可返回上一步修改影像。
- 左侧参数区：肿物类型、直径、深度、安全切缘、记录来源、病理/切缘策略。
- 中央：脸部画布，医生点击中心或勾画皮表边界。
- 右侧或底部操作区：候选切口、风险提示、长度/角度/位置微调、备选方案。
- 当前页内嵌“闭合张力模拟”，可通过按钮、抽屉或分屏打开。

完成条件：

- 肿物输入完整并记录来源。
- 系统生成候选切口。
- 医生完成必要的覆盖原因或风险确认。
- 候选进入待确认状态。

从现有功能复用：

- `/app/incision` 的肿物输入、椭圆/自由轮廓、候选生成、医生调整、候选库、审阅状态、隐私审计。
- `/app/surgery` 的闭合演示作为定性模拟组件嵌入，不作为独立主流程页面。

当前 PR 的运行时落地：

- `/app/case/:id/plan` 已提供病例内嵌“张力闭合模拟”面板，运行后把模拟状态、闭合评分、结论、摘要和更新时间保存到病例草稿。
- `/app/case/:id/evaluate` 和 `/app/case/:id/plan` 已在步骤内部加入紧凑子任务条，把“确认前置参数 / 映射张力线 / 标记病灶 / 生成候选 / 闭合模拟”呈现为临床任务，而不是工具清单。
- `/app/live`、`/app/incision` 和候选导出能力在病例步骤中以“受控评估入口 / 受控规划入口 / 受控导出入口”出现，入口旁说明进入前条件、返回后状态和临床边界。
- 病例大厅和病例步骤页共用 `ClinicalFacePreview`，在深色病例画布中显示 2D 图像 / 3D 重建 / 实时叠加模式、测量标尺、面部深度参考、解剖参考点、RSTL、病灶和切口 overlay，避免主流程停留在低保真线框脸。
- `/app/case/:id/evaluate` 的图层看板已不只是文案开关：RSTL、个性化皮纹、混合场和切口设计会直接影响病例画布；RSTL 密度、RSTL 透明度和皮纹透明度随病例草稿保存，并进入候选规则记录。
- `/app/case/:id/plan` 已把“规划依据 / 风险提示 / 审计记录”落实为 `PlanningRationalePanel`，用结构化规则卡展示年龄分档、切口模式、切缘策略、警惕区和规则记录，而不是只给最终候选线。
- `/app/case/:id/plan` 和 `/app/case/:id/review` 已增加病例内候选方案队列，候选摘要随病例草稿保存，包含版本、线性/梭形类型、长度、宽度 / 比例、尖端角、复核提示和规则记录；完整几何细调仍通过受控规划入口进入 `/app/incision`。
- 病例规划页不再把医生直接跳转到 `/app/surgery`；旧 `/app/surgery` 路由仅保留为兼容 / 研究演示入口。
- `/app/settings/atlas` 已成为图谱库管理壳，旧 `/app/annotate` 只作为图谱维护卡片中的受控入口。
- `/app/settings/developer` 已成为系统诊断壳，AI 摘要服务连接测试、三维模型预览和兼容工作台入口集中在设置中，不再直接暴露在医生主流程里。

边界要求：

- “良性/恶性”不能由系统判断。可以作为医生输入的病理或切缘策略，系统只记录并影响规则提示。
- 闭合模拟必须标注为“定性模拟 / 辅助理解”，不得暗示真实 FEM、患者个体化力学或术中规划。

### 阶段 3：方案确认与输出

定位：类似订单确认页，让医生明确将要保存或导出的方案内容。

核心内容：

- 最终切口图，包括 2D/3D 截图。
- 肿物输入摘要：类型、中心/边界、直径、深度、切缘、来源、记录者。
- 切口参数：长度、方向、宽度/比例、尖端角、平移和医生调整记录。
- 风险提示和需要临床复核的原因。
- 临床声明和合规提示。
- 导出 PDF/Markdown 报告、导出 JSON、发送到实时叠加。

完成条件：

- 审阅人已填写。
- 高风险提示已有备注或覆盖原因。
- 导出前隐私预检通过。
- 如果发送到实时叠加，必须满足 review gate。

## 文案替换原则

医生主流程不直接显示工程黑话。建议采用以下转译：

| 工程词 | 医生可见文案 |
| --- | --- |
| R3F | 不显示 |
| FLAME | 3D 面部模型 / 个体化三维面部模型 |
| MediaPipe | 面部追踪 |
| Agent trace | 规划依据 / 审计记录 |
| LLM Provider | AI 摘要服务配置 |
| topology | 头模类型 / 图谱匹配状态 |
| Guardrails | 风险提示 / 临床复核提示 |
| OpenAI-compatible / vLLM | AI 服务接口，放高级设置 |
| deterministic workflow | 规划规则流程 / 计算依据 |

例外：

- 导出 JSON、审计记录、系统诊断可以保留精确技术字段。
- 系统设置里的图谱库管理可以显示 topology、atlas、validated 等术语，但需要面向管理员解释。

## 前置参数和工具后置

UI 应先收集会影响规划的病例和病灶前提，再展示工具。

建议前置收集：

- 病例编号或匿名研究 ID。
- 年龄分档。
- 影像来源。
- 肿物类型：皮下 / 皮表。
- 肿物数据来源：触诊、超声、术前记录、医生手动输入。
- 病理/切缘策略：由医生选择或填写，不由系统判断。
- 是否允许生成候选切口草案。

工具后置：

- 线密度、透明度、镜像、调试采样点放到图层控制。
- Provider、图谱库、资产诊断放到设置。
- Trace、execution events、orchestration audit 放到审计详情。

## 医生简易操作手册映射

医生主流程的模块名称建议统一为“面部松弛皮肤张力线（RSTL）智能切口设计系统”。对外页面标题、病例大厅标题、报告标题和导出封面应使用这一名称；开发者设置和审计导出中可以保留 RSTL、atlas、provider 等技术字段。

### 患者年龄

年龄输入应在病例前置参数中收集，填写后自动分档，并把分档结果写入候选 provenance 和规划依据。第一版 UI 可以先显示规则提示；真正自动改动切口参数时，需要同步更新规则资产、候选生成逻辑、导出报告和测试。

| 年龄 | 医生可见分档 | 参数含义 |
| --- | --- | --- |
| `0-17` 岁 | 儿童 / 紧致 | 系统提示缩小梭形切口夹角；长轴:短轴建议调整为 `3.5:1`，用于应对高张力 |
| `18-59` 岁 | 中青年 / 普通 | 系统维持基础参数；当前可对应 `30°` 尖端角和 `3:1` 长轴:短轴 |
| `60` 岁及以上 | 老年 / 松弛 | 系统提示适当增加梭形切口夹角；长轴:短轴建议调整为 `2.5:1`，利用松弛度代偿 |

文案注意：医生手册中的 `1:3.5`、`1:2.5` 应在工程和 UI 中统一表达为“长轴:短轴 = `3.5:1` / `2.5:1`”，避免实现时反向理解。

### 病灶参数设置

解剖层次用下拉或分段控件呈现：

| 选项 | 激活模式 | UI 行为 |
| --- | --- | --- |
| 皮下肿物 | 线性切口模式 | 根据医生在皮肤表面描记或输入的肿物中心、轮廓类圆直径、深度和来源生成线性切口；无需设计梭形 |
| 皮表肿物 | 梭形切口模式 | 根据皮表边界、类圆化直径、安全切缘、年龄分档和局部 RSTL 方向生成梭形候选 |

切缘策略由医生选择，系统只记录和应用，不自动判断良恶性：

| 选项 | UI 行为 |
| --- | --- |
| 常规完整切除 | 默认切缘贴近病变，以完整切除为准 |
| 需扩大安全切缘 | 激活安全切缘输入框，医生手动输入扩大切除范围，例如 `5 mm`、`15 mm`；提示最终以病理切缘阴性为标准 |

### 图像与视频流采集

采集入口应落在面部评估阶段，避免和切口工具混在同一面板里：

- 上传：上传高清晰度面部照片或视频；支持正位、侧位、斜位等多角度序列，提升局部精度。
- 拍照：引导设备摄像头完成正位、侧位、斜位标准位取材。
- 3D 扫描：引导式取材并生成 3D 面部模型。
- 实时：即时人脸录制与实时扫描，用于 AR 动态跟踪显示。

### 图层看板

图层开关应面向临床概念，而不是工程模块：

| 图层 | 控制 |
| --- | --- |
| RSTL | 一键开关，支持密度和透明度调整 |
| 个性化皮纹 | 高亮额纹、鱼尾纹、鼻唇沟、睑缘纹、颏纹和其他自然褶皱凹陷 |
| 混合场 | 显示 RSTL + 个性化纹路调优结果，作为规划依据的一部分 |
| 切口设计 | 显示候选线、梭形轮廓、控制点、医生调参结果和风险提示 |

运行时要求：

- 图层开关必须改变病例画布，不能只改变表单状态。
- RSTL 至少提供低 / 标准 / 高三档密度；透明度使用可读的百分比反馈。
- 个性化皮纹透明度应独立于 RSTL，以便医生在全脸评估和局部规划之间切换关注点。
- 候选生成的规则记录应保留图层状态，例如 RSTL 密度、透明度和皮纹显示状态。

### 临床合规提示

确认页和报告导出中应强制展示完整提示；主工作台可以用固定小字号入口展示摘要。

建议文案：

> 本系统为临床辅助设计工具，不替代医生的专业判断，所有手术方案需由执业医师结合临床查体最终确认。
>
> 系统默认适应证为可直接拉拢缝合的皮肤 / 皮下肿物，超出皮肤松弛代偿范围、需皮瓣 / 植皮修复的病例，仅作切口方向参考。
>
> 恶性病灶的安全切缘需以病理检查结果为最终标准。

特殊部位提示应接入敏感结构 guardrails，医生可见文案保持直接：

> 当前病灶接近眼睑、口唇、鼻翼等警惕区，建议结合专科医生进行个性化设计。

## 病例级保存边界

病例向导不能停留在 one-time page。医生需要保存数据、划线和切口设计方案，后续才能支持草稿恢复、历史病例、医生审阅、导出和 provenance。

### 保存分层

| 层级 | 目的 | 边界 |
| --- | --- | --- |
| 本地草稿保存 | 当前浏览器会话和短期恢复 | 可用 IndexedDB / localStorage / sessionStorage；解决刷新不丢和返回微调，但不能作为正式病例记录 |
| 结构化病例保存 | 病例、肿物、标注线、候选切口、医生编辑、审阅状态、规则 trace、风险提示和 provenance | 对应 D1 或等价数据库；后端方向见 [BACKEND_DATA_ARCHITECTURE.md](BACKEND_DATA_ARCHITECTURE.md) |
| 受限资产保存 | 照片、视频、3D 头模、纹理、超声文件等敏感大文件 | 不进公开仓库，不默认进前端导出；如需在线访问，应走私有 R2 / 医院内网存储、鉴权、签名 URL 和访问审计 |

### 最小数据对象

- `case`：脱敏病例编号、状态、创建者、创建/更新时间、当前步骤。
- `patient_context`：年龄分档、病例前置参数；避免姓名、身份证、真实病历号等直接身份信息。
- `media_ref`：影像来源引用，不直接内嵌原始图片或视频 payload。
- `tumor`：解剖层次、中心、边界、直径、深度、安全切缘、数据来源、记录者。
- `annotation_lines`：医生画的 RSTL、皮纹或病灶边界线，含坐标系、单位、作者、时间和版本。
- `incision_candidate`：线性或梭形候选几何、metrics、规则依据、guardrails、候选排序和生成版本。
- `review_record`：医生确认、退回、否决、备注、覆盖原因、截图/报告导出引用和审计摘要。

当前 PR 的病例草稿已先落地轻量 `incisionCandidates` 摘要队列和 `selectedCandidateId` 指针，用于 UI 层保存候选版本、当前选中候选、规则记录和导出前审阅状态。它不是完整几何数据模型，后续接入 `/app/incision` 的候选几何和远端 API 时，应继续沿用同一字段语义扩展。

### 对 UI 重构的要求

- 病例大厅必须以 `case_id` 为核心组织草稿、历史记录和恢复入口，不能只是静态首页。
- 步骤条的每一步都应有 `saved`、`dirty`、`needs_review`、`blocked` 等状态反馈。
- 医生划线、肿物输入和切口候选都应显示“已保存 / 未保存 / 保存失败 / 可恢复草稿”。
- 返回微调不能丢失后续候选；候选应有版本链，而不是只覆盖当前 UI state。
- 导出 JSON / 报告只是输出形式，不等于系统保存；正式审阅记录应进入病例记录。
- 组件层不要直接依赖 localStorage，应通过 `DataSource` / repository 接口调用，后续可从 `LocalDataSource` 切到 Worker API。

### 隐私约束

- 不把原始人脸照片、视频帧、摄像头画面、3D 纹理、超声文件或 Provider secret 写入普通审阅 JSON。
- 公开仓库只允许示例、脱敏或合成数据。
- 每次候选生成、医生编辑、审阅确认和规则覆盖都要记录作者、时间、版本和原因。
- 真实患者数据进入任何远端存储前，需要先确定访问控制、审计、保留期限和医院/伦理边界。

## 临床级视觉设计规范

下一轮 UI 重构的视觉目标不是“好看”或“有科技感”，而是临床辅助决策系统需要的安全、精准和高效。界面应降低医生误读风险，让人脸影像、3D 模型、张力线、切口候选和风险提示成为主角。

### 设计参考

Carbon Design System 可以作为设计规范标杆，但不要求在代码层面完全替换现有技术栈。

- Carbon 适合本项目的原因：克制、严谨、适合高密度数据、企业级后台和 AI 洞察场景，并且强调无障碍和高信噪比。
- 前端可以继续使用 Tailwind / shadcn 风格组件，但需要抛弃默认 Tailwind UI 的营销感、卡片感和装饰感。
- Tailwind 只是 CSS 工具，应通过主题、token、间距、边框、色彩和组件状态写出接近 Carbon 的临床专业界面。

### 字体与 Tailwind token

临床辅助决策系统的字体目标是高压场景下快速扫视参数时不误读、不抖动、不疲劳。建议主字体使用 Neo-grotesque / 中性无衬线：

```css
:root {
  --font-clinical-sans: "IBM Plex Sans", Inter, Roboto, -apple-system, BlinkMacSystemFont, "SF Pro Text", "Segoe UI", sans-serif;
  --font-clinical-mono: "IBM Plex Mono", "SFMono-Regular", Consolas, "Liberation Mono", monospace;
}
```

Tailwind v4 可在全局 CSS 主题里暴露 token：

```css
@theme {
  --font-sans: "IBM Plex Sans", Inter, Roboto, -apple-system, BlinkMacSystemFont, "SF Pro Text", "Segoe UI", sans-serif;
  --font-mono: "IBM Plex Mono", "SFMono-Regular", Consolas, "Liberation Mono", monospace;
}
```

实现原则：

- 禁止圆润、手写、过度科技感或 display 字体进入临床工作台。
- 默认使用 `font-sans`；只有原始 ID、日志、JSON 片段、调试字段才用 `font-mono`。
- 如果不引入 Web Font，系统字体栈也可以接受；优先保证可读性和加载稳定性。
- 切口直径、深度、切缘、夹角、张力分数、表格指标、实时模拟数值必须使用等宽数字。

建议封装数字 class：

```css
.clinical-number {
  font-variant-numeric: tabular-nums lining-nums;
  font-feature-settings: "tnum" 1, "lnum" 1;
}

.clinical-number-strong {
  font-variant-numeric: tabular-nums lining-nums;
  font-feature-settings: "tnum" 1, "lnum" 1;
  font-weight: 600;
}
```

Tailwind 组件内也可以直接使用：

```tsx
<span className="tabular-nums lining-nums">12.5 mm</span>
<span className="tabular-nums lining-nums">30°</span>
<span className="tabular-nums lining-nums">86 / 100</span>
```

字号和字重建议只保留 3 到 4 个工作台梯度：

| 场景 | 建议 Tailwind | 用途 |
| --- | --- | --- |
| 模块标题 | `text-base font-semibold` | 病灶参数设置、候选切口、方案确认 |
| 正文与参数项 | `text-sm font-normal` | 患者年龄、皮表肿物、切缘策略、普通表单项 |
| 次要辅助说明 | `text-xs font-normal` | 输入框下方提示、单位说明、来源说明 |
| 警告与免责声明 | `text-xs font-medium` | 临床合规提示、特殊部位提示、导出阻断原因 |

深色模式下不能使用纯白正文。建议 token：

```css
:root {
  --clinical-text-primary: rgb(229 234 240 / 0.88);
  --clinical-text-secondary: rgb(229 234 240 / 0.66);
  --clinical-text-tertiary: rgb(229 234 240 / 0.48);
}

html {
  -webkit-font-smoothing: antialiased;
  -moz-osx-font-smoothing: grayscale;
  text-rendering: geometricPrecision;
}
```

行高建议：

- 免责声明、隐私审计、临床提示：约 `line-height: 1.5`。
- 左侧参数列表、表格、候选指标：约 `line-height: 1.2-1.3`。
- 数值表格使用固定列宽和 `.clinical-number`，不要让小数点漂移。

### 字体验收清单

- 所有 `mm`、角度、张力分数、候选指标都使用 `tabular-nums` 或 `.clinical-number`。
- 深色背景下没有纯白正文和极粗字重。
- 工作台字号以 `12 / 14 / 16 px` 为主，避免 hero-scale 字体进入病例流程。
- 警告和免责声明用 `12 px / Medium`，可见但不抢画布焦点。
- 参数列表比说明文字更紧凑，说明文字行高更舒展。
- 字体栈优先 IBM Plex Sans / Inter / Roboto / SF Pro，禁止引入圆润、手写或装饰字体。

### 整体基调：临床级精准

评价标准应从“好不好看”改为“是否干扰医生判断”。

- 去装饰化：去掉无意义渐变、发光特效、大面积彩色背景、弹跳动效和装饰性插画。
- 高信噪比：数据、影像、3D 模型和候选切口是主体，UI 框架应退居幕后。
- 操作反馈要明确，但不夸张；成功、警告和失败状态应稳定、可预测。
- 不使用游戏化或消费产品式激励语言，不把候选生成包装成“智能魔法”。

### 色彩系统：极度克制，功能导向

建议以中性灰阶为基础。可以优先考虑深色模式，因为深色背景更容易凸显面部模型、RSTL 线、切口线和局部警示，也更接近部分影像工作站的长时间观察环境。

语义色必须严格限定：

| 颜色角色 | 使用范围 |
| --- | --- |
| 红 / 橙 | 仅用于安全切缘警告、眼睑/口唇/鼻翼等警惕区、导出阻断和高风险复核 |
| 绿 | 仅用于处理成功、审阅通过、闭合模拟通过状态或明确安全反馈 |
| 青 / 高亮黄 | 仅用于画布上的 RSTL、个性化皮纹、切口候选、控制点和局部高亮 |
| 中性灰 | 普通按钮、表单、面板、分隔线、导航和非风险状态 |

禁止事项：

- 不把红/橙/绿/青这类高显著颜色用于普通导航、普通取消/确定按钮或装饰图标。
- 不使用大面积品牌渐变、彩色背景带、发光边框或漂浮色块作为页面风格。
- 不让警示色与切口线/RSTL 线混用，避免医生把普通图层误读为风险。

### 排版：数据可读性第一

- 数值字段必须优先可比较：切口直径、深度、切缘、角度、长度、张力数值等应使用等宽数字或 tabular figures，保证小数点和单位对齐。
- 单位必须贴近数值，例如 `15.0 mm`、`30°`，不要把单位藏到说明文。
- 标题、正文、辅助说明、合规提示的字号层级应清晰，但避免 hero-scale 大字进入工作台。
- 合规提示应固定可见或在确认页强制展示；主工作区可使用较小字号和低对比度，不抢夺画布焦点。
- 风险提示文案要短、直接、可操作；详细解释进入展开区或审计详情。

### 空间与布局：高密度面板

医疗工作台的信息密度天然较高，不应盲目追求大留白。

- 使用紧凑型网格和稳定列宽，参数区、候选区、审阅区和报告区应便于横向对比。
- 面板之间优先用 1px 细线、弱底色和明确分组标题区分，不用厚重阴影制造层级。
- 卡片圆角保持克制，避免“电商卡片墙”或“效率工具看板”的轻浮感。
- 画布周边控件应最少化，图层开关可以浮动，但不能遮挡病灶、候选切口或敏感结构提示。
- 主按钮只服务当前步骤的唯一下一步动作；次要操作放入工具栏或更多菜单。

### 组件状态

- 普通按钮使用中性灰、描边或弱填充。
- 主按钮只用于“下一步”“确认方案”“生成报告”等流程推进动作。
- 风险状态必须有颜色、文字和图标三重表达，不能只依赖颜色。
- Disabled 状态需要说明原因，例如“请先完成面部映射”或“高风险提示需填写备注”。
- Hover、focus 和 keyboard navigation 需要满足无障碍要求；医生可能在触控板、鼠标、键盘和大屏之间切换。

### 设计验收清单

- 主流程页面没有无意义渐变、发光、弹跳动画、大面积彩色背景或装饰插画。
- 普通导航和普通按钮没有使用红/橙/绿/青等语义色。
- 风险色只出现在风险提示、警惕区、导出阻断或安全切缘相关场景。
- 数值字段使用等宽数字或 tabular figures，并明确显示单位。
- 参数面板采用紧凑网格、细线分隔和弱底色，不依赖大阴影。
- 3D/2D 画布是视觉主体，侧栏和浮层不遮挡临床判断对象。
- 合规提示可见但不抢主视觉；确认页必须强制展示完整合规声明。
- 所有警告和 disabled 状态都有文字解释，不能只靠颜色。

## 设计风险与避坑约束

病例向导式重构不是把工具页简单串起来。后续 UI 设计和实现需要把下面四类风险作为验收约束。

### 不要把向导式流程做成单向锁死

风险表现：如果评估、定位、规划、闭合模拟必须线性完成，完成 A 才能看 B，回退后丢失后续设计，医生会失去专家控制感。

临床现实：医生判断通常是网状和迭代式的。例如在切口模拟阶段发现张力线显示不够清楚，需要临时回到面部评估阶段调整 RSTL 密度或图层透明度。

落地要求：步骤条应该是“状态可溯”的面包屑导航，而不是一次性流程锁。系统底层可以按阶段组织数据，但操作体验上必须允许无缝返回微调，并保留已生成的候选方案、标注和审阅状态。

### UI 方案不能脱离前端状态架构

风险表现：设计稿如果直接把 `/app/live`、`/app/incision`、`/app/surgery` 融合成一个全局工作台，但没有对应的状态提升和数据生命周期设计，会出现跨页面数据断层、模型重复加载、闭合模拟拿不到切口候选等问题。

技术现实：病例数据、影像输入、肿物轮廓、切口候选、闭合模拟结果和审阅记录都需要进入统一的 case/session state。

落地要求：UI 设计交付时需要同时标注状态流转：哪些数据跨步骤保留，哪些可撤销，哪些需要重新计算，哪些只是视图配置。前端实现上应继续坚持 React UI + Zustand / typed snapshot / renderer bridge 的边界，不要让 3D 高频状态进入普通 React state。

### 极简界面不能造成黑盒 AI 信任危机

风险表现：为了追求“高级感”，只展示最终候选切口线，而隐藏 guardrails、张力分数、面部分区、年龄参数、切缘规则等依据，会让医生觉得系统是黑盒。

临床现实：医生不会盲从一个自动生成的面部手术切口。系统需要把“为什么这么画”说清楚，哪怕默认折叠，也要随时可查看。

落地要求：切口候选旁必须有“规划依据 / 风险提示 / 审计记录”入口。内容可以用短语和结构化数据呈现，例如 `高张力区 -> 缩小夹角`、`60 岁及以上 -> 长宽比调整为 2.5:1`、`接近眼睑警惕区 -> 建议专科复核`。

### 3D/AR 画布区域必须控制视觉抢占和性能成本

风险表现：在画布上叠加过多半透明面板、悬浮微动效、密集网格或复杂 React overlay，会抢夺医生对面部模型、RSTL、切口线的注意力，也可能拖慢实时 AR 追踪。

技术现实：FLAME/3D 头模、实时摄像头、MediaPipe/视觉追踪、切口曲线和闭合模拟本身已经占用 GPU/主线程资源。

落地要求：3D 画布应是主角，UI 覆盖层要轻、少、稳定。避免复杂阴影、磨砂玻璃、大面积透明叠层和持续动画；图层控制、参数调节、闭合模拟结果尽量放在固定面板或轻量 overlay 中。实时模式下至少关注模型加载、相机开启、图层切换、切口拖拽和闭合模拟这几个场景的帧率与交互延迟。

## 技术落地建议

第一轮重构只做 UI 壳和路由重组，不重写几何算法。

建议新增：

- `CaseWorkflowShell`：病例向导外壳，包含病例头、步骤条、主画布区域和主操作按钮。
- `CaseStepper`：评估、定位、规划、确认步骤。
- `CaseHeader`：病例编号、状态、返回病例大厅。
- `ClinicalCanvas`：承载 live/incision/surgery runtime 的统一画布容器。
- `LayerControls`：张力线、密度、透明度、辅助线索等图层开关。
- `PlanningPanel`：肿物输入、候选结果、医生微调和闭合模拟入口。
- `ReviewSummary`：最终方案确认和导出。
- `caseStore`：病例级低频状态，保存当前步骤、影像来源、肿物输入、候选摘要和审阅状态。
- `CaseSaveStatus`：统一展示已保存、未保存、保存失败、可恢复草稿和导出完成状态。
- `CaseDataSource`：收敛病例、肿物、标注、候选和审阅记录的读写接口，先接 `LocalDataSource`，后续可切到 Worker API。
- `PlanningRationalePanel`：折叠展示规划依据、风险提示、guardrails、医生覆盖原因和审计记录。

复用现有边界：

- React / Zustand 仍只管理低频 UI 状态。
- Three.js、MediaPipe、canvas、逐帧 landmarks 和 mesh buffers 继续留在 runtime/controller 内。
- 当前 `liveRuntime`、`incisionAgentRuntime`、`SurgeryR3FScene` 先作为可嵌入模块复用。
- `DataSource` / repository 层负责保存与恢复，不允许组件分散直接写 localStorage。

## 迁移顺序

1. 新增病例大厅和步骤条外壳，把现有 `/app/live`、`/app/incision` 作为步骤内嵌入口。
2. 把 `/app/surgery` 改造成可嵌入的闭合模拟面板，同时保留旧路由兼容。
3. 把 `/app/annotate` 从主导航移到设置/图谱库管理。
4. 隐藏 `/app/three-preview`，只在开发者设置中保留。
5. 把 Provider 配置、资产状态和工程诊断迁到设置或审计详情。
6. 统一医生主流程文案，替换技术词。
7. 接入病例级 `CaseDataSource` 和保存状态反馈，先实现本地草稿，再对齐后端 API。
8. 最后再逐步消化 legacy controller，避免 UI 重构和核心几何重构同时发生。

## 验收清单

- 医生从病例大厅进入，不再直接面对技术路线列表。
- 主流程用步骤条串联：面部评估 -> 标记病灶 -> 切口规划 -> 方案确认。
- 病例步骤内部用子任务条表达当前阶段的临床动作；旧画布入口必须被包装成受控入口，并说明进入前、返回后和边界。
- 病例大厅和病例步骤页必须使用统一的高对比临床画布组件，至少包含视图模式、测量标尺、解剖参考、病灶和切口 overlay；不能退回只有脸型轮廓的低保真示意图。
- `/app/live`、`/app/incision`、`/app/surgery`、`/app/annotate` 和 `/app/three-preview` 不出现在医生首页主导航中；旧工具只能从病例步骤上下文或系统设置中的受控入口进入。
- Provider 配置不出现在切口规划主侧栏。
- 闭合模拟在切口规划上下文中打开，并显示定性模拟边界。
- 主流程文案不出现 R3F、FLAME、MediaPipe、Agent trace、topology。
- 导出和审计仍保留技术 provenance，不丢失可追溯性。
- 病例大厅、步骤条和候选列表都围绕 `case_id`、保存状态和候选版本组织。
- 病例规划页和方案确认页必须能显示病例内候选方案队列，候选至少包含版本、当前选中状态、长度 / 宽度 / 比例 / 尖端角、复核提示和规则记录。
- 年龄分档、病灶层次、切缘策略、图层看板和临床合规提示覆盖医生简易操作手册。
- 图层看板必须包含 RSTL 开关、RSTL 密度、RSTL 透明度、个性化皮纹开关/透明度、混合场和切口设计开关，并且这些状态会反映到病例画布和本地病例草稿。
- 候选切口旁有结构化规划依据 / 风险提示 / 审计记录，至少展示年龄参数、病灶层次、切缘策略、警惕区和规则记录，不把 AI 或规则判断做成黑盒。
- 数值字段、角度、张力分数和候选 metrics 使用等宽数字或 `tabular-nums`。
- 现有 `/app/live`、`/app/incision`、`/app/annotate`、`/app/surgery` 路由在过渡期仍可回归测试；`/app/annotate` 和 `/app/three-preview` 只能从设置或兼容区进入。
- 视觉冒烟检查可用 `cd web && npm run visual:case` 启动本地 Vite，并用 Playwright 截取病例大厅、评估、规划、确认、图谱库管理和系统诊断页；截图输出到 `local_outputs/case-workflow-visual/`，不提交到仓库。若运行环境缺少 Chromium 依赖，应在 PR 中记录失败日志和人工预览链接；若 Linux 截图中文字缺失，需要在截图主机安装 Noto Sans CJK SC 等中文字体后再做设计截图复核。

## 非目标

- 不在本轮重写几何 kernel、切口候选算法或 RSTL 方向服务。
- 不把闭合模拟升级为临床级 FEM。
- 不在本轮实现远端病例数据库、权限系统或电子签名；但 UI 和组件边界必须预留 `DataSource` / repository 接口。
- 不删除科研标注工具或系统诊断，只从医生主流程隐藏。
- 不声明任何候选方案可直接用于临床执行。
