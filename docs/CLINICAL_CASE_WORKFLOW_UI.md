# 临床病例工作流 UI 重构方案

本文记录下一轮 UI 层重构的产品化方向：从按工具/技术模块组织的工作台，转向按病例任务组织的临床向导式工作流。目标是让医生围绕一个病例完成评估、定位、规划、模拟和输出，而不是面对一组研发工具自行拼流程。

## 目标

- 医生主流程只暴露临床任务：面部评估、病灶定位、切口规划、闭合模拟、方案确认。
- 标注器、图谱库、Provider 配置、R3F 预览、资产加载和工程诊断从医生主流程移出。
- 现有功能以隐藏、重组、嵌入为主，不直接删除，避免破坏研发、科研和回归测试路径。
- 页面文案去技术化，不向医生显示 R3F、FLAME、MediaPipe、Agent trace、topology 等工程术语。
- 用步骤条和病例状态代替左侧工具堆叠，明确每一步的输入、完成条件和下一步动作。

## 当前功能重新定位

| 当前路径 / 功能 | 当前状态 | 产品化定位 |
| --- | --- | --- |
| `/app/live` 实时 RSTL 显示 | Stage 1 主路径 | 纳入“面部评估与张力线映射”，作为病例向导第一步 |
| `/app/incision` 切口候选工作台 | Stage 2 主路径 | 纳入“病灶定位与切口规划”，作为病例向导第二步 |
| `/app/surgery` 沿 RSTL 闭合演示 | 独立演示页 | 降级为“闭合张力模拟”功能按钮/抽屉，嵌入切口规划步骤 |
| `/app/annotate` 3D 线标注 | 图谱生产/复核工具 | 移出医生主流程，放入系统设置或图谱库管理 |
| `/app/three-preview` R3F 标准脸预览 | 技术验证页 | 默认隐藏，只保留开发者诊断入口 |
| LLM Provider 配置 | 远端 OpenAI-compatible / vLLM 连通性测试 | 移到全局设置或开发者配置，不放在医生主画布旁 |
| 浏览器 workflow trace / Agent trace | 审计与工程诊断 | 默认隐藏在“规划依据 / 审计记录”里，必要时展开 |
| FLAME / MediaPipe / topology 状态 | 资产和拓扑工程状态 | 转译为“3D 面部模型 / 面部追踪 / 图谱匹配状态” |

## 信息架构

产品化后建议分成三类入口：

| 类型 | 面向对象 | 放置位置 |
| --- | --- | --- |
| 临床主流程 | 医生 | 病例大厅和病例向导 |
| 科研/管理工具 | 研究员、管理员 | 系统设置、图谱库管理、模型/规则配置 |
| 工程诊断 | 开发者 | 隐藏入口、诊断面板、DevTools、审计导出 |

建议新增或重组为以下路由层级：

| 路由建议 | 职责 |
| --- | --- |
| `/app/cases` | 病例大厅，显示新建病例、导入病例、草稿和历史记录 |
| `/app/case/new` | 新建病例入口，收集基础病例信息和前置参数 |
| `/app/case/:id/evaluate` | 面部评估与张力线映射 |
| `/app/case/:id/plan` | 病灶定位、候选切口规划和闭合模拟 |
| `/app/case/:id/review` | 方案确认、报告导出和实时叠加输出 |
| `/app/settings/atlas` | 图谱库管理，承载当前 3D 标注工具 |
| `/app/settings/developer` | Provider、资产、诊断和隐藏技术预览 |

现有 `/app/live`、`/app/incision`、`/app/annotate`、`/app/surgery` 可以先保留为兼容入口或 redirect，不需要在第一轮重构中删除。

## 病例向导流程

### 阶段 0：病例大厅

定位：医生进入系统后的工作台首页，不直接进入人脸画布。

主要内容：

- 新建面部评估 / 导入新病例。
- 最近病例、草稿、已确认方案。
- 每个病例显示状态：待评估、待标记病灶、待确认、已导出。
- 全局设置入口：图谱库管理、AI 服务配置、开发者诊断。

设计原则：

- 不展示 R3F、FLAME、MediaPipe、Provider 等工程概念。
- 不把科研标注工具放在医生主按钮区。
- “继续上次病例”优先于“选择技术路线”。

### 阶段 1：面部评估与张力线映射

定位：确认当前病例的影像或 3D 输入足够支撑张力线显示。

布局建议：

- 左侧：病例基础信息、年龄分档、影像来源、采集方式。
- 中央：照片、视频、摄像头或 3D 面部画布。
- 右上浮层：图层开关，例如基础张力线、线密度、透明度、个性化皮纹追踪。
- 底部或右下：主按钮“下一步：标记病灶”。

完成条件：

- 已上传照片/视频、打开摄像头，或完成可用的 3D 面部输入。
- 面部追踪状态达到可用。
- 张力线映射成功，或系统明确提示需要重新采集。

从现有功能复用：

- `/app/live` 的上传、摄像头、暂停、导出、线密度、透明度、镜像、追踪质量。
- 切口实时叠加 QA 继续保留，但默认转译为“方案叠加状态”。

### 阶段 2：病灶定位与切口规划

定位：把关注点从全脸切换到局部病灶和候选切口。

布局建议：

- 顶部：步骤条和病例摘要，可返回上一步修改影像。
- 左侧参数区：肿物类型、直径、深度、安全切缘、记录来源、病理/切缘策略。
- 中央：脸部画布，医生点击中心或勾画皮表边界。
- 右侧或底部操作区：候选切口、风险提示、长度/角度/位置微调、备选方案。
- 当前页内嵌“闭合张力模拟”，可通过按钮、抽屉或分屏打开。

完成条件：

- 肿物输入完整并记录来源。
- 系统生成候选切口。
- 医生完成必要的覆盖原因或风险确认。
- 候选进入待确认状态。

从现有功能复用：

- `/app/incision` 的肿物输入、椭圆/自由轮廓、候选生成、医生调整、候选库、审阅状态、隐私审计。
- `/app/surgery` 的闭合演示作为定性模拟组件嵌入，不作为独立主流程页面。

边界要求：

- “良性/恶性”不能由系统判断。可以作为医生输入的病理或切缘策略，系统只记录并影响规则提示。
- 闭合模拟必须标注为“定性模拟 / 辅助理解”，不得暗示真实 FEM、患者个体化力学或术中规划。

### 阶段 3：方案确认与输出

定位：类似订单确认页，让医生明确将要保存或导出的方案内容。

核心内容：

- 最终切口图，包括 2D/3D 截图。
- 肿物输入摘要：类型、中心/边界、直径、深度、切缘、来源、记录者。
- 切口参数：长度、方向、宽度/比例、尖端角、平移和医生调整记录。
- 风险提示和需要临床复核的原因。
- 临床声明和合规提示。
- 导出 PDF/Markdown 报告、导出 JSON、发送到实时叠加。

完成条件：

- 审阅人已填写。
- 高风险提示已有备注或覆盖原因。
- 导出前隐私预检通过。
- 如果发送到实时叠加，必须满足 review gate。

## 文案替换原则

医生主流程不直接显示工程黑话。建议采用以下转译：

| 工程词 | 医生可见文案 |
| --- | --- |
| R3F | 不显示 |
| FLAME | 3D 面部模型 / 个体化三维面部模型 |
| MediaPipe | 面部追踪 |
| Agent trace | 规划依据 / 审计记录 |
| LLM Provider | AI 摘要服务配置 |
| topology | 头模类型 / 图谱匹配状态 |
| Guardrails | 风险提示 / 临床复核提示 |
| OpenAI-compatible / vLLM | AI 服务接口，放高级设置 |
| deterministic workflow | 规划规则流程 / 计算依据 |

例外：

- 导出 JSON、审计记录、开发者诊断可以保留精确技术字段。
- 系统设置里的图谱库管理可以显示 topology、atlas、validated 等术语，但需要面向管理员解释。

## 前置参数和工具后置

UI 应先收集会影响规划的病例和病灶前提，再展示工具。

建议前置收集：

- 病例编号或匿名研究 ID。
- 年龄分档。
- 影像来源。
- 肿物类型：皮下 / 皮表。
- 肿物数据来源：触诊、超声、术前记录、医生手动输入。
- 病理/切缘策略：由医生选择或填写，不由系统判断。
- 是否允许生成候选切口草案。

工具后置：

- 线密度、透明度、镜像、调试采样点放到图层控制。
- Provider、图谱库、资产诊断放到设置。
- Trace、execution events、orchestration audit 放到审计详情。

## 技术落地建议

第一轮重构只做 UI 壳和路由重组，不重写几何算法。

建议新增：

- `CaseWorkflowShell`：病例向导外壳，包含病例头、步骤条、主画布区域和主操作按钮。
- `CaseStepper`：评估、定位、规划、确认步骤。
- `CaseHeader`：病例编号、状态、返回病例大厅。
- `ClinicalCanvas`：承载 live/incision/surgery runtime 的统一画布容器。
- `LayerControls`：张力线、密度、透明度、辅助线索等图层开关。
- `PlanningPanel`：肿物输入、候选结果、医生微调和闭合模拟入口。
- `ReviewSummary`：最终方案确认和导出。
- `caseStore`：病例级低频状态，保存当前步骤、影像来源、肿物输入、候选摘要和审阅状态。

复用现有边界：

- React / Zustand 仍只管理低频 UI 状态。
- Three.js、MediaPipe、canvas、逐帧 landmarks 和 mesh buffers 继续留在 runtime/controller 内。
- 当前 `liveRuntime`、`incisionAgentRuntime`、`SurgeryR3FScene` 先作为可嵌入模块复用。

## 迁移顺序

1. 新增病例大厅和步骤条外壳，把现有 `/app/live`、`/app/incision` 作为步骤内嵌入口。
2. 把 `/app/surgery` 改造成可嵌入的闭合模拟面板，同时保留旧路由兼容。
3. 把 `/app/annotate` 从主导航移到设置/图谱库管理。
4. 隐藏 `/app/three-preview`，只在开发者设置中保留。
5. 把 Provider 配置、资产状态和工程诊断迁到设置或审计详情。
6. 统一医生主流程文案，替换技术词。
7. 最后再逐步消化 legacy controller，避免 UI 重构和核心几何重构同时发生。

## 验收清单

- 医生从病例大厅进入，不再直接面对技术路线列表。
- 主流程用步骤条串联：面部评估 -> 标记病灶 -> 切口规划 -> 方案确认。
- `/app/annotate` 和 `/app/three-preview` 不出现在医生主导航中。
- Provider 配置不出现在切口规划主侧栏。
- 闭合模拟在切口规划上下文中打开，并显示定性模拟边界。
- 主流程文案不出现 R3F、FLAME、MediaPipe、Agent trace、topology。
- 导出和审计仍保留技术 provenance，不丢失可追溯性。
- 现有 `/app/live`、`/app/incision`、`/app/annotate`、`/app/surgery` 路由在过渡期仍可回归测试。

## 非目标

- 不在本轮重写几何 kernel、切口候选算法或 RSTL 方向服务。
- 不把闭合模拟升级为临床级 FEM。
- 不实现病例数据库、权限系统或电子签名。
- 不删除科研标注工具或开发者诊断，只从医生主流程隐藏。
- 不声明任何候选方案可直接用于临床执行。
